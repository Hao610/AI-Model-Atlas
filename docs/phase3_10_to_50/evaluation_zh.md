# 模型评估与测试 📏

[[English] (evaluation.md)](evaluation.md) | [中文]

当你修改了提示词（Prompt）、更换了底层向量数据库，或是微调了模型之后，你怎么知道 AI 的回答质量到底是变好了还是变差了？

传统软件可以通过单元测试（Unit Test）判断代码是否跑通。但 AI 的输出是主观的语言文本，无法用简单的对错判定。为了衡量模型实力，我们需要引入**模型评估方法**。

---

## 🏆 四大核心大模型评测法

| 评测方法 | 运作机制 | 最佳适用场景 | 优缺点对比 |
| :--- | :--- | :--- | :--- |
| **BLEU / ROUGE** | 计算 AI 输出的词汇与标准人类答案之间的**字词重合度**。 | 机器翻译、文章摘要。 | ⚡ **优点**：计算速度极快，完全免费。<br>⚠️ **缺点**：过于死板，如果 AI 用了同义词（如“开心”代替“愉快”）就会被判低分。 |
| **人工评测 (Human Eval)** | 招募真实的领域专家或测试人员，对照打分表给 AI 的回答挨个打分。 | 高风险业务（如医疗、法务咨询）或高要求的产品上线前终测。 | ⚡ **优点**：极度精准，最符合人类直觉。<br>⚠️ **缺点**：非常昂贵，速度极其缓慢，无法自动化。 |
| **竞技场盲测 (Chatbot Arena)** | 将两个模型（如模型 A 和模型 B）隐去名字。输入相同问题，由人类根据回答盲测投票（计算类似游戏天梯的 Elo 分数）。 | 通用基础模型能力排名。 | ⚡ **优点**：客观公平，真实反映大众口碑。<br>⚠️ **缺点**：不适合企业内部私有数据的测试。 |
| **大模型裁判 (LLM-as-a-Judge)** | 让一个逻辑能力最强的模型（如 GPT-4o 或 Claude 3.5）去阅读用户问题、参考文档和候选模型的回答，并给候选模型打分（1~10分）。 | 企业 RAG 知识库系统、智能客服系统。 | ⚡ **优点**：速度极快，可以完全自动化，成本低，适合跑大规模测试集。<br>⚠️ **缺点**：裁判模型可能存在偏见（比如倾向于给文风接近自己的模型打高分）。 |

---

## 🤖 深度解密：大模型裁判 (LLM-as-a-Judge) 工作流

这是目前工业界评估 RAG 知识库系统时最通用的自动化测试方案（例如使用知名框架 **Ragas**）：

```text
       【 用户问题 】 + 【 检索出来的上下文文档 】
                           │
                           ▼
                  【 待测试模型的回答 】
                           │
                           ▼
          ┌──────────────────────────────────┐
          │      裁判大模型 (如 GPT-4o)       │
          │ - 对照原始问题与上下文材料。       │
          │ - 从以下维度对回答进行打分：       │
          │   1. 忠实度 (Faithfulness，有无胡编)│
          │   2. 回答相关性 (Relevance)       │
          │   3. 幻觉程度 (Hallucination)    │
          └────────────────┬─────────────────┘
                           │
                           ▼
                   【 自动评分卡 (1-10分) 】
```

在实际开发中，我们在代码上线前，会在后台自动对 100 道历史测试题跑一遍这个流程。如果综合评分没有下降，我们才敢放心地把新代码推送到生产环境。
