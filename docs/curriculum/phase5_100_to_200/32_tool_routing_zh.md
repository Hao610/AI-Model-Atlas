# 模块 32: 智能体路由 (Tool Routing)

[[English] (32_tool_routing.md)](32_tool_routing.md) | [中文]

欢迎来到 AI 工程的下一个层级。你已经构建了一个基础的 RAG (检索增强生成) 管道，模型可以通过获取相关文档来回答问题。这是一个很好的开始。但是，如果用户提出的问题需要实时数据、复杂的数学计算，或是编写和运行代码，会发生什么呢？

标准的 RAG 管道将会失败，因为它只知道如何搜索其静态数据库。

现在，你该如何将它变成一个精心编排的系统呢？这就是 **Agentic RAG (智能体 RAG)** 和 **Tool Routing (工具路由)** 登场的时候了。

> **核心洞察：** AI 不再是模型，而是系统。 (AI is no longer just a model, it's a system.)

## 为什么 ChatGPT 不仅仅是一个模型

当你与 ChatGPT 或 Claude 等现代 AI 产品交互时，你并不是在与一个单一的、庞大的语言模型对话。你正在与一个经过精心编排的生态系统进行交互。

如果你询问最新的天气，它会搜索网络；如果你要求它绘制图表，它会编写 Python 代码并在安全的沙箱中执行。语言模型本身并不直接进行数学计算或网络抓取——它扮演着**路由器 (Router)** 的角色，决定*应该使用哪个工具*。

这是该架构工作原理的简化视图：

```text
User (用户)
  │
  ▼
[ Router (LLM) ] ──▶ (需要最新数据？) ──▶ [ Search ]
  │
  ├──▶ (需要精确计算？) ──────────────────▶ [ Calc ]
  │
  └──▶ (需要数据分析/执行？) ─────▶ [ CodeRunner ]
  │
  ▼
(工具执行结果返回)
  │
  ▼
[ LLM 合成最终答案 ]
  │
  ▼
User (用户)
```

## 工具路由是如何工作的

在 Agentic RAG 中，语言模型被赋予了一个“工具箱”（可用函数或 API 的列表），以及每个工具的功能描述。

当用户提示输入时，其过程如下：
1. **分析 (Analyze)：** LLM 读取提示并评估它是可以直接回答，还是需要工具的帮助。
2. **路由 (Route)：** 如果需要工具，LLM 会输出结构化命令（如 JSON），指定工具名称和要传递给它的参数。
3. **执行 (Execute)：** 系统拦截此命令，运行外部工具（例如查询 API），并捕获输出结果。
4. **合成 (Synthesize)：** 工具的输出被反馈到 LLM，然后 LLM 读取结果并为用户制作自然、准确的回复。

## 从单一管道到自主智能体

通过实现工具路由，你的工作从构建静态的“管道”晋级为构建“智能体 (Agents)”。

你的 AI 不再是一个被动的百科全书，它是一个主动的问题解决者：能够进行计划、从多个来源收集信息、与外部软件交互，并合成复杂的答案。作为一名 AI 系统工程师，你的职责从单纯的提示词调试转变为设计那些你的智能体将要挥舞的、强大且可靠的工具。
