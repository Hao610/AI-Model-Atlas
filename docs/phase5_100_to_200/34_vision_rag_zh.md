# 模块 34: 视觉解析 (Vision RAG & OCR)

[[English] (34_vision_rag.md)](34_vision_rag.md) | [中文]

---

你打造的 RAG 系统在处理纯文本时堪称完美。当你问它关于公司历史的问题时，它能对答如流。但是，当你上传一份企业财务报告（PDF），并问一个简单的问题：“2024年第三季度的营收是多少？”时，系统却开始胡言乱语，或者直接说不知道。

为什么你的 RAG 系统会在企业 PDF 上翻车？因为它把表格摧毁了。是时候给你的 RAG 系统装上“眼睛”了。

## 核心洞察：表格不是文本，是结构

大多数简单的文档加载器和 OCR 工具将页面视为扁平的词语序列。当遇到表格时，它们会从左到右、逐行读取，完全无视赋予数字意义的行与列。

**表格是 RAG 中最困难的部分。**

### 结构的毁灭：视觉演示

想象一个简单的财务表格：

| 指标 | 2024 Q3 | 2024 Q4 |
|---|---|---|
| 营收 (Revenue) | 150M | 200M |
| 支出 (Expenses)| 100M | 120M |

**普通分块器是如何读取它的（文本毁灭）：**
`指标 2024 Q3 2024 Q4 营收 150M 200M 支出 100M 120M`

一旦被压扁成这个字符串，“营收”、“2024 Q3”和“150M”之间的对应关系就彻底消失了。如果用户问“Q4 的支出是多少？”，大语言模型（LLM）看到的是 `200M 支出 100M`，它会陷入混乱，甚至产生幻觉。

**Vision RAG 系统是如何看待它的（结构化布局）：**
Vision RAG 系统会保留边界框以及表格的 HTML 或 Markdown 表示。

```html
<table>
  <tr><th>指标</th><th>2024 Q3</th><th>2024 Q4</th></tr>
  <tr><td>营收</td><td>150M</td><td>200M</td></tr>
  <tr><td>支出</td><td>100M</td><td>120M</td></tr>
</table>
```

## 如何给你的 RAG 系统装上眼睛

现代 Vision RAG 流程不再盲目提取文本，而是使用多模态模型或高级文档解析工具来理解页面的**布局**。

1. **布局检测 (Layout Detection)：** 系统首先识别 PDF 上的区域：“这是一个段落”、“这是一张图片”、“这是一个表格”。
2. **表格提取 (Table Extraction)：** 专门的模型（如 Table Transformer）提取表格并将其转换为结构化格式（Markdown 或 HTML）。
3. **多模态嵌入 (Multi-Modal Embeddings)：** 一些先进的系统直接使用视觉-语言模型嵌入表格的图像，保留所有视觉上下文。
4. **摘要增强 (Summary Augmentation)：** 我们可以将结构化表格传递给 LLM，让它编写表格的纯文本摘要，并将摘要和原始表格一起存储在向量数据库中。

通过尊重文档的视觉布局，我们就能避免在 LLM 尚未读取数据之前就将其破坏。
