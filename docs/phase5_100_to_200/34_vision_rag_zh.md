# 第 34 模块：视觉 RAG 与 OCR (Vision RAG & OCR)

传统的检索增强生成 (RAG) 管道在处理 PDF 等富文本文档格式时经常会遇到困难。虽然标准的文本提取工具可以捕获段落文本，但它们在处理**表格 (Tables)** 和**图像 (Images)** 等结构化或视觉元素时，往往会遭遇灾难性的失败。视觉 RAG 和光学字符识别 (OCR) 技术通过实现对复杂视觉布局的提取、理解和分块，弥补了这一差距。

## 解析 PDF 的挑战

PDF 本质上是关于在画布上何处绘制字符和线条的指令；它们缺乏语义结构（例如 HTML 标签）。这给 RAG 系统带来了两个主要挑战：

1. **表格：** 标准解析器通常将表格逐行或逐列提取为混乱的文本流。表头和单元格值之间的对应关系丢失，导致 LLM 无法准确回答基于数据的查询。
2. **图像和图表：** 包含在图表、流程图或信息图中的重要信息，如果使用基础文本提取器，通常会被完全跳过或转录得很糟糕。

如果这些视觉元素被简单地忽略或破坏，生成的向量嵌入将是不完整或具有误导性的，从而严重降低 RAG 系统的准确性。

## 表格感知分块 (Table-Aware Chunking)

为了保留表格数据的语义完整性，我们使用一种称为**表格感知分块 (Table-Aware Chunking)** 的技术。

与纯粹按照字符数或段落换行符对文档进行分块不同，表格感知分块会检测表格的边界。它确保将整个表格（或其合理的子集以及表头）保留在同一个块 (chunk) 中。或者，它可以将表格转换为 Markdown 或 HTML 格式，然后再进行向量化嵌入，从而保留行/列结构，以便 LLM 在生成阶段能够正确“阅读”表格。

## 利用 `pdfplumber` 和 `PyMuPDF`

我们利用强大的 PDF 处理库来防止数据碎片化：

* **`pdfplumber`：** 非常适合精确、程序化地提取表格。它分析线条和字符的交点以重建表格网格。通过使用 `pdfplumber`，我们可以准确识别表格的边界框，干净地提取单元格内容，并将其格式化为 Markdown 表格。这保证了表格数据不会被分割成多个不连贯的文本块。
* **`PyMuPDF` (fitz)：** 一个高性能库，用于渲染页面和提取图像。对于视觉 RAG，我们使用 `PyMuPDF` 提取图表或复杂视觉块的高分辨率图像。然后，这些图像可以由视觉语言模型 (VLM) 或 OCR 引擎（如 Tesseract 或云端 OCR API）进行处理，以生成描述性文本摘要，最后将其嵌入并添加到向量存储中。

通过结合用于结构化表格的 `pdfplumber` 和用于视觉元素的 `PyMuPDF`，我们的 RAG 管道从“仅文本”升级为“文档感知”，从而解锁包含在复杂报告和论文中的完整知识。
