# 模块 35: 知识图谱 (GraphRAG)

[[English] (35_graph_rag.md)](35_graph_rag.md) | [中文]

想象一下，你正在调查一起复杂的企业欺诈案，或者在数以千计的医学期刊中研究一种罕见疾病。在这些场景中，仅仅匹配文本段落是不够的。你需要知道“公司 A”拥有“子公司 B”，而“子公司 B”向“高管 C”转移了资金。你正在处理一个庞大的法律或医疗数据网络，在这个网络中，发现实体之间的联系比匹配文本更重要。

这就是 GraphRAG（基于知识图谱的检索增强生成）发挥作用的地方。这里的核心洞察简单而深刻：**“关系比内容更重要。”** (Relationships are more important than content.)

## 什么是 GraphRAG？

传统的 RAG 系统将文档切分成块，并基于语义相似度进行检索。它们非常擅长回答“手册里关于 X 是怎么说的？”，但在回答“X、Y 和 Z 在我们整个公司历史中是如何关联的？”时却表现得很糟糕。

GraphRAG 会从你的数据中构建一个知识图谱——一个由节点（实体）和边（关系）组成的网络。当用户提出问题时，系统会遍历这个图谱来检索相关联的事实，从而为 LLM 提供结构化的上下文，而不仅仅是一系列文本块。

## 你什么时候真正需要它？

> [!WARNING]
> **GraphRAG Is Not Magic (GraphRAG 不是魔法)**
> 
> GraphRAG solves structure problems, not knowledge problems. (它解决的是结构问题，而不是知识问题。) 它对于像法律判例、医学研究或复杂的供应链分析等关系决定答案的特定领域非常有效。然而，业界宣称“GraphRAG = 下一代 RAG”能取代一切的炒作是错误的。对于 90% 的普通文档检索任务来说，混合检索 (Hybrid Search) + RRF (倒数秩融合) 通常就足够了，而且成本更低，也更容易维护。

当你的问题是这样时，请使用 GraphRAG：
*   “零号病人与新爆发的疫情之间有哪些间接联系？”
*   “与这起法律纠纷相关的公司的所有董事会成员是谁？”

**不要**在以下情况使用 GraphRAG：
*   “总结这份 PDF。”
*   “我该如何重置密码？”

## 它是如何工作的（简化版）

1.  **实体抽取：** LLM 阅读你的原始文档并提取实体（人物、组织、疾病、地点）。
2.  **关系映射：** LLM 识别这些实体是如何连接的（例如，“人物 A”*是*“组织 B”的*CEO*）。
3.  **图谱构建：** 这些实体和关系被存储在图数据库（如 Neo4j）中。
4.  **图谱检索：** 当提出查询时，系统会找到相关的起始实体，并遍历它们的连接以构建全面的上下文窗口。
5.  **生成：** LLM 使用这个映射出来的网络来回答问题。

GraphRAG 非常强大，但构建和维护知识图谱需要大量的工程努力。只有当数据点之间的关系蕴含着信息的真正价值时，才选择使用它。
