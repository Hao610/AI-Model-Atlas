# AI 工作流架构 🔄

[[English] (13_ai_workflows.md)](13_ai_workflows.md) | [中文]

开发一个真正的 AI 应用，绝不仅仅是把一段提示词直接扔给大模型那么简单。你需要设计一套有条不紊的**工作流 (Workflow)**，在这个流程中，大模型只是诸多步骤中的一环。

本章我们用一张直观的工作流架构图，来拆解一个典型的“Agent + RAG”系统的内部运行机制。

---

## 🗺️ 完整的 AI 系统架构流向图

以下是用户提问在专业级 AI 应用中是如何被处理的完整步骤：

```text
【1. 用户输入】 ──► “请问我们公司第三季度的销售目标是多少？”
       │
       ▼
【2. 智能体核心 (Agent Core)】
  - 拆解分析用户的请求意图。
  - 判断出该问题需要调取内部事实文档。
       │
       ▼
【3. 检索模块 (RAG)】
  - 将问题转化为向量坐标。
  - 去向量数据库 (如 Chroma) 中检索。
  - 捞取匹配度最高的退货政策/销售计划段落。
       │
       ▼
【4. 提示词拼装与上下文注入】
  - 将：系统人设指令 + 捞取出来的段落原文 + 用户原问题 组合封装成大段文本。
       │
       ▼
【5. 大语言模型 (LLM) 推理】
  - 阅读刚才拼装好的上下文。
  - 组织语言，撰写出基于事实的、安全的回答。
       │
       ▼
【6. 安全沙箱与内容拦截 (Guardrails)】
  - 检查模型输出是否包含敏感词、隐私数据或错误代码。
       │
       ▼
【7. 最终输出给用户】 ──► “我们公司第三季度的销售目标是 50,000 美元……”
```

---

## ⚡ 核心组件功能拆解

1. **用户输入 (User Input)**：用户在网页聊天框里输入的原始指令。
2. **智能体核心 (Agent Core)**：系统的路由中枢。在高级系统里，它就像一个调度员，决定：*“这个问题我是自己回答，还是去查知识库，或者去跑一段 Python 脚本计算一下？”*
3. **检索模块 (RAG)**：搜索雷达。它负责把人类语言翻译成机器坐标，在向量数据库中找出语义高度重合的段落卡片，并把它搬到大模型的“书桌”上。
4. **上下文注入 (Context Ingestion)**：把搜出来的事实论据，与用户的问题打包成大模型能看懂的“材料包”。
5. **大模型推理 (LLM Inference)**：大脑引擎。它负责阅读“材料包”，用它极强的自然语言表达能力，把材料总结提炼出通顺的回答。
6. **安全护栏 (Guardrails)**：安全阀。用来监测 AI 的输出，防止 AI 发生幻觉胡说八道、泄露敏感 API 密钥或使用不当言辞。

---

## 🎨 为什么需要设计工作流？

如果不设计工作流，你只是在碰运气，期盼 AI 能记住你教过它的所有事实。而设计了工作流之后：
* 你能严格控制 AI 在回答问题时参考了哪几页文件。
* 你能将 AI 的回答范围限制在你的企业数据库中，防止它天马行空。
* 你能大幅节省 API 费用——因为你只把与问题最相关的几段话喂给 AI，而不是每次聊天都把整本用户手册上传上去。
