# 推理优化与高并发服务 🏎️

[[English] (inference_optimization.md)](inference_optimization.md) | [中文]

在线上生产环境中运行大语言模型，绝非仅仅把模型文件塞进显存那么简单。如果有 100 个用户同时调用你的接口，如何保证 GPU 不会排队卡死？如何让模型打字速度更快？

本章将带你拆解大模型高并发推理（LLM Inference Optimization）的三大核心技术：**KV Cache**、**动态批处理 (Continuous Batching)** 以及 **流式输出 (Streaming)**。

---

## 💾 1. KV Cache (键值缓存)

大模型在输出文本时，是“逐字（Token）”生成的。为了生成第 5 个字，它必须把前 4 个字读入并进行数学矩阵计算。如果没有优化，每生成一个新字，大模型都要把前面已经生成过的所有历史文字**重新计算一遍**，这会导致计算量呈 $O(N^2)$ 指数级减慢。

**KV Cache（键值缓存）** 完美解决了这一重复计算痛点：
* 模型在生成前面的文字时，把计算好的中间数学状态（Key 和 Value 矩阵）保存在显存里。
* 生成下一个新字时，它只计算最新那个字的数学状态，之前的字直接从显存缓存中拉取。

*代价与折中：KV Cache 虽然极大加快了打字速度，但它会吞噬大量的 GPU 显存。显存被缓存占满后，模型能同时处理的并发用户数就会受限。*

---

## 📦 2. 动态批处理 (Continuous Batching)

在传统深度学习里，我们会把请求攒成一个 Batch（批次）一起送给 GPU 计算。但大模型生成文本的长度是随机的。如果用户 A 只需要 1 个字的简短回答，而用户 B 需要一篇 500 字的作文，传统的 Batch 会强制让用户 A 坐在那干等，直到用户 B 的作文全部打完，GPU 才会释放这一批任务。

**动态批处理 (Continuous Batching)**（由 vLLM 等推理引擎发扬光大）打破了这一枷锁：
* 它的调度细粒度到了 **Token 级**，而不是请求级。
* 一旦用户 A 的单字回答生成完毕，该插槽在 Batch 中会**立刻被释放**并分配给排队的用户 C，而此时用户 B 的作文仍在继续生成，GPU 的核心计算单元始终保持 100% 满载运转。

```text
传统批处理：【 用户 A (1字已完工，干等) 】【 用户 B (500字打字中) 】 ──► 显卡必须等两者都完工才释放
动态批处理：【 用户 A 完工 ──► 瞬间换上 用户 C 】                      ──► 显卡零等待，并发率极高
```

---

## ⚡ 3. 流式输出 (Streaming Output)

对于前端用户来说，最直接影响体验的指标是 **首字延迟 (Time-To-First-Token, TTFT)**——即我敲下回车后，屏幕上多快能吐出第一个字。

* **标准 JSON 模式**：服务器必须默默等待大模型把整整 500 字的段落全部打完，然后把巨大的 JSON 包通过网络发送给前端。用户在网页前会面对 10 秒钟的菊花等待转圈。
* **流式输出模式 (Streaming)**：使用 **SSE (Server-Sent Events)** 技术，模型每生成一个字，服务器就**在微秒级时间内**把这个字推送到用户的浏览器。用户看着 AI 像打印机一样实时打字，即使总生成时间相同，用户体验也呈指数级提升。

---

## 📊 概念总结：延迟 vs. 吞吐量

* **延迟 (Latency)**：单个用户请求得到回复的速度（主要通过 KV Cache 和流式输出进行极致优化）。
* **吞吐量 (Throughput)**：服务器每分钟能同时处理多少个用户的并发提问（主要通过动态批处理和模型量化来榨干 GPU 性能）。
